ID: SEC-005
Title: Path Traversal Vector in Attachments Route
Severity: High
CVSS: CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:N/A:N (6.5)
Category: A01:2021 Broken Access Control / CWE-22
Status: Suspected (needs runtime verification)

Location:
- file: apps/server/src/http.ts
- function: attachmentsRouteLayer
- lines: 102

Affected entry points:
- GET /attachments/* -> apps/server/src/http.ts:102 -> File System access (line 120)

Description:
The application serves files from the attachments directory based on the URL path. It attempts to prevent path traversal by calling a normalization function `normalizeAttachmentRelativePath`. However, relying on custom string manipulation and checks for `..` can often be bypassed using URL encoding (e.g. `%2e%2e%2f`) or unexpected path segment normalization.

Vulnerable code:
```typescript
const rawRelativePath = url.value.pathname.slice(ATTACHMENTS_ROUTE_PREFIX.length);
const normalizedRelativePath = normalizeAttachmentRelativePath(rawRelativePath);
```

Attack scenario:
1. Attacker sends a request to `/attachments/..%2f..%2f..%2fetc%2fpasswd`.
2. If `normalizeAttachmentRelativePath` fails to decode or correctly identify the traversal, it might resolve to `/etc/passwd`.
3. The application serves the sensitive file.

Proof of concept:
Send the following curl request:
```bash
curl "http://localhost:8080/attachments/..%2f..%2f..%2f..%2fetc%2fpasswd"
```

Impact:
Local File Inclusion (LFI) leading to sensitive data disclosure.

Root cause:
Custom path traversal prevention instead of standard library checks.

Remediation:
Always use `path.resolve()` and verify that the final resolved path starts exactly with the intended base directory.

```typescript
const finalPath = path.resolve(config.attachmentsDir, rawRelativePath);
if (!finalPath.startsWith(path.resolve(config.attachmentsDir) + path.sep)) {
    return HttpServerResponse.text("Invalid attachment path", { status: 400 });
}
```
