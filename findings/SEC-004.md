ID: SEC-004
Title: Predictable Randomness / Weak Cryptography in HMR Key
Severity: Low
CVSS: CVSS:3.1/AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:N/A:N (3.7)
Category: A02:2021 Cryptographic Failures / CWE-330
Status: Confirmed

Location:
- file: apps/web/src/components/ComposerPromptEditor.tsx
- function: Global scope
- lines: 80

Affected entry points:
- React component key generation -> apps/web/src/components/ComposerPromptEditor.tsx:80

Description:
The application uses `Math.random()` to generate a "unique" key for Hot Module Replacement (HMR): `const COMPOSER_EDITOR_HMR_KEY = "composer-editor-" + Math.random().toString(36).slice(2);`. `Math.random()` is not a cryptographically secure pseudorandom number generator (CSPRNG) and produces predictable values.

Vulnerable code:
```typescript
const COMPOSER_EDITOR_HMR_KEY = `composer-editor-${Math.random().toString(36).slice(2)}`;
```

Attack scenario:
While currently only used for an HMR key, an attacker could theoretically predict the state of the PRNG. If `Math.random()` is used elsewhere for security-sensitive operations in the same context, it could compromise them.

Proof of concept:
An attacker observes a sequence of keys and predicts the next key:
```javascript
const nextKey = predictNextRandom(Math.random());
```

Impact:
Low risk in the current context, but establishes a bad practice.

Root cause:
Use of `Math.random()` instead of `crypto.randomUUID()`.

Remediation:
Use `crypto.randomUUID()`:
```typescript
const COMPOSER_EDITOR_HMR_KEY = `composer-editor-${crypto.randomUUID()}`;
```
